# Enterprise Official Site Context

## 目标

在官网相关修改中统一品牌叙事、信息口径、页面策略与文案风格，避免每次改动出现定位漂移。

## 触发条件

- 用户要求修改官网页面结构、导航、模块内容、CTA
- 用户要求修改公司介绍、品牌故事、产品矩阵、口号
- 用户要求修改需求文档、公司背景文档、实施规划
- 用户要求新增或优化官网视觉与交互风格描述

## 执行步骤

1. 在开始修改前，优先读取以下文档并提炼关键口径：
   - `docs/公司背景文档.md`
   - `docs/需求文档.md`
   - `docs/技术实现文档.md`
   - `.trae/skills/official-site-context/plan.md`
2. 检查当前改动是否保持以下一致性：
   - 公司基础信息与对外主体信息一致
   - 产品矩阵命名与层级一致
   - 面向家长/孩子的话术边界一致
3. 风格基线采用以下参考：
   - `https://www.apipost.cn/`
   - 对齐方向：清晰的信息分区、明确的核心价值主张、突出的行动按钮、模块化内容编排
4. 输出时必须包含：
   - 修改后信息摘要
   - 与既有口径差异点
   - 可用于后续实现的结构化字段

## 文案规范

- 首屏优先回答“是谁、解决什么问题、下一步做什么”
- 每个页面仅保留一个主目标动作
- 避免空泛形容词，优先“能力 + 场景 + 结果”
- 对儿童教育相关内容保持审慎和正向表达

## 质量检查

- 是否覆盖品牌、产品、转化、合规四类信息
- 是否存在成立年份、注册地址、经营范围口径冲突
- 是否输出了可直接进入实现阶段的字段与模块清单

## 规则

- 每次有新的修改时，都需要重新执行该技能，以确保修改与品牌叙事、信息口径、页面策略与文案风格保持一致。
- 每次修改后，都要检查《需求文档》《技术实现文档》（以及项目根 README.md）是否与修改内容一致，并同步更新文档
- 每次大的 commit 提交后，酌情考虑是否是有必要记录，维护一套《开发者日志》与 《plan.md》（即 plan 开发计划文档），记录每次修改的详细信息。
- 每次修改后，都要检查《plan.md》是否与修改内容一致，并同步更新文档
